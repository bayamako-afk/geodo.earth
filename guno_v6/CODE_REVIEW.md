# GUNO V6 コードレビューレポート

**作成日**: 2026-03-07  
**作成者**: Manus AI  
**対象範囲**: GUNO V6 (Phase 1 〜 Phase 6) の全マージ済みコード

---

## 1. 全体アーキテクチャ評価

GUNO V6 は、V5 のモノリシックな構造（`guno_v5.js` への依存）から脱却し、**「純粋なゲームロジック」と「UI・通信層」を明確に分離した優れたアーキテクチャ**を実現しています。

特に以下の点が優れています：

- **単方向データフロー**: `User Action → Engine → State → UI Render` の流れが徹底されており、状態管理が非常にクリーンです。
- **純粋関数の徹底**: `src/core/` 以下のモジュール（`game_engine.js`, `rules.js`, `scoring.js`）は DOM や外部 API に一切依存しておらず、テスト駆動開発（TDD）が容易な構造になっています。
- **Host-Authoritative モデル**: オンライン同期において、ホストが状態の真実（Source of Truth）を持ち、ゲストはアクションのみを送信する設計は、チート対策や状態の不整合を防ぐ上で最適な選択です。

## 2. モジュール別レビュー

### 2.1. データ層 (`src/data/pack_loader.js`)

**【評価】**
V5 の複雑なパース処理を整理し、`hub_bonus_deck` と `hub_bonus_global` を明確に分離した点は、引き継ぎ書の要件を完全に満たしています。

**【改善提案】**
- **エラーハンドリング**: `loadPackFromUrl` で `fetch` が失敗した場合のエラーメッセージは親切ですが、JSON パースエラー（SyntaxError）時の捕捉がありません。`try-catch` でラップし、より分かりやすいエラーメッセージを返すことを推奨します。
- **型定義の明文化**: JSDoc で `PackData` などの構造が記述されていますが、将来的に TypeScript への移行（または JSDoc での `@typedef` の充実）を行うと、さらに保守性が向上します。

### 2.2. コアエンジン層 (`src/core/`)

**【評価】**
`game_engine.js` と `rules.js` の分離は完璧です。特に `isPlayable` 関数が純粋関数として切り出されているため、UI のヒント表示（`getPlayableIndices`）とバックエンドのバリデーションの両方で安全に再利用できています。

**【改善提案】**
- **CPU AI の拡張性**: 現在の `runCpuTurn` は「出せるカードがあれば出す」という非常にシンプルなロジックです。将来的に「GUNO リーチの路線を優先する」「他プレイヤーの妨害を優先する」などの戦略を組み込めるよう、Strategy パターンを導入することをお勧めします。
- **イベント名の定数化**: `emit({ type: "card_played" })` のようにマジックストリングが使用されています。`const EVENTS = { CARD_PLAYED: "card_played", ... }` のように定数化するとタイポを防げます。

### 2.3. 通信層 (`src/net/`)

**【評価】**
Supabase Realtime の `postgres_changes` と `broadcast` を巧みに組み合わせています。特に `version` カラムを用いた楽観的ロック（Optimistic Locking）は、競合状態を防ぐ堅牢な設計です。

**【改善提案】**
- **再接続処理（リコネクト）**: `TransportSupabase` クラスにおいて、ネットワーク切断時の自動再接続ロジックが実装されていません。Supabase クライアントは内部的に再接続を試みますが、UI 側に「再接続中...」を表示する仕組みがあるとユーザー体験が向上します。
- **ゲストのアクションバリデーション**: 現在、ホストは `handleGuestAction` でゲストのアクションをそのまま `playCard` に渡しています。`playCard` 内で `isPlayable` チェックが行われるため安全性は保たれていますが、悪意のあるゲストが不正な `cardIndex`（配列外など）を送信した場合の防御的プログラミング（`cardIndex` の範囲チェックなど）を追加するとより安全です。

### 2.4. UI層 (`src/ui/` & `main.js`)

**【評価】**
V5 のデザイントークンを継承しつつ、コンポーネント指向（`hand.js`, `board.js`, `log.js`）でファイルを分割したことで、可読性が劇的に向上しています。

**【改善提案】**
- **`main.js` の肥大化**: 現在 `main.js` が 450 行を超えており、オンライン対戦のロジック（`handleOnlineGameStart` など）も含まれています。これを `src/app/local_game.js` と `src/app/online_game.js` のように責務ごとに分割すると、さらに見通しが良くなります。
- **レンダリングの最適化**: `renderAll()` は状態が変わるたびに DOM を再構築します。現在の規模ではパフォーマンスに問題はありませんが、将来的にアニメーションやエフェクトを多用する場合、Virtual DOM 的なアプローチ（または差分更新のみ行う仕組み）の導入を検討する余地があります。

## 3. セキュリティとインフラ（Supabase）

**【評価】**
Row Level Security (RLS) が適切に設定されています。特に `rooms` テーブルの UPDATE/DELETE をホストのみに制限している点は優れています。

**【改善提案】**
- **古いルームのクリーンアップ**: 現在 `rooms` テーブルの有効期限切れルームは、クライアント側で `listRooms` 時にフィルタリングして非表示にしているだけです。Supabase の Edge Functions と pg_cron を組み合わせて、定期的に古いルームを物理削除するバッチ処理を実装することを推奨します。
- **匿名セッションの限界**: `localStorage` に保存した `guno_v6_session_id` に依存しているため、ユーザーがブラウザのキャッシュをクリアするとホスト権限を失います。本格的なリリース時には、Supabase Auth を用いた匿名ログイン（Anonymous Sign-ins）機能への移行を検討してください。

## 4. 総評

GUNO V6 は、V5 の複雑さを解消し、モダンでスケーラブルな Web アプリケーションとして見事に生まれ変わりました。コードの品質は非常に高く、引き継ぎ書の要件をすべて満たしています。

現在のコードベースは、将来的な機能追加（マッチメイキング、レーティングシステム、拡張パックの追加など）に十分耐えうる堅牢な土台となっています。上記の改善提案は、本格的なプロダクションリリースに向けた「次のステップ」としてご活用ください。
