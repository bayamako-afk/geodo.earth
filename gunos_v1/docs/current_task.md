# GUNOS V1.3 Task 03 — Mobile Portrait Layout Toggle

## Objective
V1.3 Task 01〜02 で整備した **Map-first / HUD overlay** レイアウトを前提に、  
**スマホ縦表示（mobile portrait）での視認性・操作性を改善**する。

本タスクでは、通常のPC表示や既存のAUTOプレイ挙動を壊さず、  
スマホ縦時のみ HUD の見せ方を最適化する。

狙いは、  
**「狭い縦画面でも、地図が主役のまま遊べること」**  
である。

---

## Background
Task 01 では、分割パネル型から **Map-first / HUD overlay** 構造へ移行した。  
Task 02 では、HUD全体を半透明・blurベースで統一し、  
GUNOSらしい “地図上に浮かぶ情報UI” の質感を整えた。

しかし、スマホ縦表示では

- ヘッダー
- 下部手札エリア
- ステータス表示
- 補助情報

が同時に存在すると、地図の可視領域が圧迫されやすい。

そこで Task 03 では、**スマホ縦だけ情報量を整理するレイアウト切替**を導入する。

---

## Design Goal
目指す状態は以下。

- スマホ縦でも地図中心が見える
- HUDが重なりすぎない
- 手札操作がしやすい
- 情報は必要十分に絞られる
- 「管理画面」ではなく「地図ゲーム」に見える
- PC / タブレット / 横画面は極力維持する

---

## Core Idea
スマホ縦表示では、**常時表示する情報を減らし、必要なものだけを前面化する**。

つまり、

- 常時主役：地図
- 常時必要：手札
- 補助情報：コンパクト化 or 折りたたみ化
- 二次情報：必要時のみ確認

という優先順位にする。

---

## Layout Requirements

### 1. Mobile Portrait Detection
以下の条件を主対象とする。

- スマホ縦表示
- 目安幅: `max-width: 680px`
- 必要なら height 条件も併用可

この条件下のみ、専用レイアウトを適用する。

---

### 2. Header Simplification
スマホ縦では `#app-header` の情報量を減らす。

要件:
- ヘッダー高さを縮小する
- 上下余白を詰める
- 文字サイズや内部gapを少し圧縮してよい
- ただし Start / Auto / 主要操作は隠さないこと
- 圧縮しても押しにくくしないこと

目的:
- 上端の圧迫感を減らす
- 地図の可視高さを増やす

---

### 3. Bottom Hand Area Optimization
手札エリアはスマホ縦でも **下中央の主操作HUD** として維持する。

要件:
- 画面下中央配置を維持
- 高さを可能な限り薄くする
- 余白・padding・gapをスマホ向けに圧縮
- 手札の視認性とタップ性は維持
- 必要ならカードサイズを微調整してよい
- 地図を過度に隠さないこと

重要:
- 「見えるけど押しにくい」はNG
- 「押しやすいけど厚すぎて地図が死ぬ」もNG
- バランス重視

---

### 4. Status / Score Information Compaction
ステータスやスコア系の情報は、スマホ縦では **常時大型表示しない**。

推奨方針:
- 右上に小型HUD化
- 1行要約化
- アイコン＋短い数値化
- 必要なら詳細を折りたたみ表示にする

例:
- 現在ターン
- 山札枚数
- 都市名
- 簡易スコア

ただし、情報が読めなくなるほど削りすぎないこと。

---

### 5. Secondary Panel Toggle
ログや補助情報など、主役でない情報は  
**モバイル縦では折りたたみ / 開閉式 / 最小化** を優先する。

方針:
- 常時大きく置かない
- 小さなボタン・タブ・チップで展開可能でもよい
- とにかくプレイ中の地図領域を優先する

---

### 6. Center Map Visibility
スマホ縦で最も守るべきことは、**地図中央の可視領域**である。

要件:
- 上下HUDに挟まれても中央の地図が見える
- プレイ中に「地図を見ている感覚」が失われない
- 結果オーバーレイ以外は中央を占有しない

---

## Visual Style Requirements
モバイル縦専用調整でも、Task 02 のデザイン方針を維持する。

- 半透明背景
- 軽いblur
- 角丸
- 細い境界線
- 地図を潰さない軽さ

避けること:
- 不透明な巨大パネル
- 情報を詰め込みすぎた管理画面風UI
- モバイル時だけ別アプリのように見えること

---

## Technical Constraints
- ゲームロジックは変更しない
- 既存の overlay 構造を維持する
- CSS中心の対応を優先する
- 必要なら最小限のclass追加は可
- pointer-events 制御を壊さないこと
- Leaflet map のパン・ズーム操作を損なわないこと

---

## Preferred Implementation Direction
以下の優先順で進めること。

1. スマホ縦専用の header 圧縮
2. bottom hand area の薄型化
3. status/score のコンパクト化
4. secondary info の最小化 or toggle化
5. 390px幅前後で微調整
6. プレイ動作確認

---

## Expected Deliverables
以下を提出すること。

1. 実装結果
2. 変更ファイル一覧
3. モバイル縦向けに何を変えたかの要約
4. スクリーンショット
   - PC表示
   - スマホ縦表示（例: 390px幅）
   - 必要ならスマホ横表示
5. 確認結果
   - 地図操作可能
   - 手札操作可能
   - Start / Auto 操作可能
   - AUTOプレイ継続可能
   - 勝敗表示に問題なし
   - コンソールエラーなし

---

## Acceptance Criteria
以下を満たせば完了とする。

- スマホ縦で地図の中心領域がしっかり見える
- 手札が操作しやすい
- ヘッダーが圧迫しすぎない
- ステータス表示が整理されている
- 二次情報が常時主張しない
- PC表示を壊していない
- V1.3 Task 01〜02 の Map-first HUD 方針を維持している

---

## Notes
このタスクの本質は、単なるレスポンシブ対応ではない。

**「小さい画面でも、地図が主役であり続ける」**
ことが目的である。

情報を全部見せるのではなく、  
地図体験を守るために **情報の優先順位を整理する** こと。

GUNOSは管理UIではなく、  
あくまで **地図ベースのネットワークゲーム** として見えることを重視する。
