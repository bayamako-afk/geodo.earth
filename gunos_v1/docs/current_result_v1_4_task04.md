# GUNOS V1.4 Task 04 Completion Report

## 概要
GUNOS V1.4 Task 04「Score Reason Breakdown」の実装が完了しました。本タスクでは、各ターンにおけるスコア変動の理由を短く視覚化し、プレイヤーが「なぜ今のスコアが増減したのか」を直感的に理解できる軽量なUIを追加しました。これにより、GUNOSが「駅と路線の価値を点数化して競うゲーム」であることがより明確に伝わるようになりました。

## 実装内容の詳細

### 1. Score Reason Breakdown UI の追加
右上スコアサマリーパネル（`#map-overlay-scores`）の直下に、新たに `#score-reason-panel` を追加しました。このパネルは以下の要素で構成されています。
- **理由タグ（Reason Tags）**: スコア変動の要因を示す短いバッジ。最大3件まで表示されます。
- **短い理由テキスト（Short Reason Line）**: 変動の主な理由を1行で説明するミュートカラーのテキスト。

### 2. スコア変動の検出と理由生成（score_reason.js）
新しく作成した `score_reason.js` モジュールにて、前ターンからのスコア差分（Delta）を検出し、それに基づいた理由タグを生成するロジックを実装しました。
- **Hub ボーナス**: `hub_bonus` が増加した場合、その増加量に応じて `+N Hub★` / `+N Hub+` / `+N Hub` のタグを生成します。
- **Route+ ボーナス**: `route_bonus` が増加した場合、`+N Route+` タグを生成します。
- **駅取得 / ルート完成**: `station_score` が増加した場合、ルート完成時であれば `+N Capture`、それ以外は `+N Score` タグを生成します。
- **リード差の変化**: プレイヤー間のスコア差（Gap）が変動した場合、トッププレイヤーが差を広げれば `Lead↑`、追うプレイヤーが差を縮めれば `Lead+` タグを生成します。

### 3. 既存の Meaning System との統合
Task 01〜03 で整備した視覚的な意味体系（Meaning System）と完全に互換性を持たせました。
- 理由タグには、Hub系を示す `◆` と、Route+系を示す `▶` のアイコンプレフィックスを使用しています。
- タグのカラーパレットやボーダー、発光効果（glow）も、既存の Station Hint や Candidate Indicator と完全に一致させています。

## レスポンシブ対応と Map-first 方針の維持
詳細な採点表を常設するのではなく、「理由の見える化」に特化した軽量な設計を採用し、地図の視認性（map-first）を維持しました。
- **デスクトップ**: 理由タグと1行の理由テキストが完全に表示されます。
- **モバイル縦（Portrait）**: 画面幅を考慮し、理由テキストを非表示（`display: none`）にし、タグのフォントサイズを縮小、表示数を最大2件に制限しました。
- **モバイル横（Landscape）**: 地図の縦幅を最大限に確保するため、理由テキストを非表示にし、タグのサイズと余白を極限まで切り詰めました（最大2件）。

## テスト結果
ローカルサーバー環境にて、以下の動作を確認しました。
- ゲーム開始直後（Turn 0）およびスコア変動がないターンでは、パネルが完全に非表示（`opacity: 0`）になること。
- スコアが変動したターンに、該当する理由タグ（例：`◆ +12 Hub★`, `+57 Score`, `Lead↑`）と理由テキスト（例："Hub bonus increased this turn."）がフェードインして表示されること。
- PC、モバイル縦、モバイル横の各ビューポートにおいて、表示が破綻せず、意図した通りの省略（Truncation）が行われること。
- コンソールエラーが発生せず、既存のゲーム進行やスコア計算に影響を与えないこと。
